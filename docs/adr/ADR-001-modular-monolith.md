# ADR-001: Выбор Modular Monolith Architecture

## Контекст
Необходимо выбрать архитектурный стиль для системы корпоративной отчётности "Відлік", которая должна поддерживать:
- 10,000+ пользователей
- Сложные бизнес-процессы согласования
- Потенциальный переход к микросервисам в будущем
- Высокие требования к безопасности и аудиту

## Рассматриваемые варианты

### 1. Monolithic Architecture (Classic)
- **Плюсы:** Простота разработки, развёртывания, тестирования
- **Минусы:** Сложность масштабирования, жёсткая связанность, риск "big ball of mud"

### 2. Microservices Architecture
- **Плюсы:** Независимое масштабирование, технологическая гибкость
- **Минусы:** Сложность разработки и эксплуатации, распределённые транзакции, network latency

### 3. Modular Monolith (Выбран)
- **Плюсы:** 
  - Чёткие границы доменов (Bounded Contexts)
  - Возможность эволюционного перехода к microservices
  - Простота транзакционной целостности (ACID в рамках одного DB)
  - Легче в отладке и тестировании
- **Минусы:** Требует дисциплины в разделении модулей

## Решение
Выбрана **Modular Monolith** архитектура с чётким разделением на домены:
- Auth, Users, Reports, Tasks, Approvals, Notifications, Audit, Analytics, AI, Exports
- Каждый домен — отдельный npm-пакет/директория с собственными:
  - Entities/Repositories
  - Services
  - Controllers
  - DTOs
  - Events

## Последствия
- Положительные:
  - Упрощённый CI/CD pipeline
  - Единая точка входа (API Gateway)
  - Простая миграция данных
  - Централизованный мониторинг
- Отрицательные:
  - Необходимость строгой дисциплины при разработке
  - Ограниченная горизонтальная масштабируемость отдельных доменов
